# Internationalization (i18n) — design

Date: 2026-07-08
Status: approved (brainstorm), pending implementation plan

## Motivation

The desktop app ships English-only. All user-facing UI strings are hardcoded in
JSX across ~59 component/route files (~123 string literals: JSX text plus
`placeholder`/`title`/`aria-label` attributes). There is no i18n layer.

The app targets scientists globally. Two audiences justify translation: **major
research nations** (large user base) and **countries with strong social-media
reach** (organic spread on X/Twitter etc.). The barrier today is not translation
volume — it is small — but the lack of a design that keeps strings out of the
code. Once strings are extracted into a single, typed, lint-guarded layer,
adding a language becomes cheap: translate one set of JSON files.

## Scope

**In scope — the frontend UI shell only.** Every user-facing string rendered by
the React side: menus, buttons, labels, placeholders, tooltips, empty states,
settings, and toast/inline messages originating in the frontend.

**Explicitly out of scope**

- **AI-generated content** — agent replies, artifacts, notebook bodies. These
  follow the model's output, not the UI locale; i18n cannot and must not touch
  them.
- **Rust-side strings** (phase 1) — a handful of user-facing error/status strings
  surface via toasts (e.g. `"Jupyter is not set up yet"`, Modal auth hints).
  They stay English in phase 1. A future seam (error codes → frontend
  translation) is noted but not built now.
- **Demo project prose** — `examples/bci-trends/` copy stays as authored.

## Goals

- Extract 100% of frontend UI strings into a typed, namespaced i18n layer.
- Make adding a language a translate-only task (no code changes).
- Ship a first batch of four languages, all LTR.
- Build the **RTL seam** now so Arabic (and other RTL languages) is a cheap,
  well-scoped second batch — not a rewrite.
- Prevent regression: new hardcoded UI strings must fail lint; missing
  translations must fail a test.

## Non-goals (deferred)

- Arabic translation and the full RTL visual audit (second batch — see §7).
- Rust-side and demo-project translation.
- Translating AI-generated content.
- Traditional Chinese (`zh-Hant`) and other languages beyond the first batch —
  the structure supports them; they are added on demand.

## Language set

**First batch (this design):**

| Locale     | Language            | Direction | Rationale |
|------------|---------------------|-----------|-----------|
| `en`       | English             | ltr       | Source of truth / fallback; global lingua franca |
| `zh-Hans`  | Simplified Chinese  | ltr       | Top-tier research output; home market |
| `ja`       | Japanese            | ltr       | Research strength + one of X/Twitter's largest markets |
| `es`       | Spanish             | ltr       | One language covers Spain + Latin America; high social reach |

**Registered but not yet shipped:** `ar` (Arabic, `dir: rtl`) — kept in the
locale registry with its direction, excluded from the selectable list until the
second batch completes.

**Structure reserves** (added on demand later): `de`, `fr`, `ko`, `pt-BR`.

## Approach

Chosen: **react-i18next with typed resources** (evaluated against a hand-rolled
`t()` and LinguiJS). It is the industry-standard, gives plural/interpolation/
lazy-loading for free, uses translator-friendly JSON, and supports compile-time
key typing (a missing key is a TypeScript error). It best serves the core goal —
strings never scatter again — while staying a well-understood, low-risk choice.

Dependencies: `i18next`, `react-i18next`, `i18next-browser-languagedetector`.

## Design

### 1. Layout

```
apps/desktop/src/i18n/
  index.ts                 # i18next init, detection, lazy-load config
  resources.d.ts           # type augmentation: key autocomplete + missing-key errors
  locales/
    en/{common,settings,runs,session,inspector,errors}.json   # source of truth
    zh-Hans/{...}.json
    ja/{...}.json
    es/{...}.json
```

Namespaces are feature-aligned so each JSON file stays small and scannable:
`common` (shared actions/labels), plus one per major area (`settings`, `runs`,
`session`, `inspector`, `errors`). Start with these; split further only if a
file grows unwieldy.

### 2. Key convention

Dot-namespaced, semantic (not English-derived): `common.actions.save`,
`settings.language.label`, `runs.empty.title`. Interpolation via
`t('runs.count', { count })`; plurals via i18next `_one`/`_other` suffixes.

### 3. Usage

```tsx
const { t } = useTranslation('settings')
// ...
{t('language.label')}
```

Components reference **keys only** — never literal copy. `en` JSON is the source
of truth and the authoritative key set.

### 4. Detection & switching

- **Resolution order:** stored preference → OS/browser locale (Tauri/`navigator`)
  → `en` fallback. Detected locale is nearest-matched to a supported one.
- **Manual override:** a language selector in Settings, alongside the theme
  control. Preference persists in the existing `useUiStore` (which already owns
  `theme`) via a new `locale` field and its persistence mechanism.
- **Side effects on change:** update `<html lang>` and `<html dir>` from the
  active locale's registry entry. `dir` is `ltr` for all first-batch languages;
  the mechanism is live so RTL flips correctly once enabled.

### 5. Number & date formatting

Route through `Intl` / i18next formatters bound to the active locale. Replace the
existing hardcoded spots (`toLocaleString("en-US")`, `toLocaleDateString(undefined, …)`)
so dates and numbers follow the chosen locale. In-scope cleanup.

### 6. Migration

1. **Mechanical extraction** — move the ~123 existing strings into `en/*.json`,
   replacing literals with `t()` calls. UI behavior is unchanged and verifiable
   at this step (English still renders identically).
2. **Translate** — produce `zh-Hans`, `ja`, `es` from the `en` key set.

### 7. RTL readiness (seam now, Arabic later)

Direction is a **global** concern, never per-component:

- `<html dir>` is driven once from the active locale's `direction`; children
  inherit. **No component hardcodes direction or takes a `dir` prop.**
- New/changed components use **logical CSS**: Tailwind `ps-/pe-/ms-/me-/start-/
  end-/text-start/text-end` instead of physical `pl-/pr-/ml-/mr-/left-/right-/
  text-left`. Directional icons (arrows, chevrons) mirror via the `rtl:` variant
  or a transform.
- `ar` is registered with `dir: rtl` but excluded from the selectable list.

**Second batch (deferred):** Arabic translation + sweep of all ~59 components
converting physical→logical direction classes + RTL visual QA, then enable `ar`.
The seam makes this "translate + rename classes," not a rewrite.

### 8. Regression guards

- **Lint:** `eslint-plugin-i18next` `no-literal-string` rejects new hardcoded JSX
  strings — strings cannot scatter back into the code.
- **Test:** a unit test asserts every non-`en` locale's key set is **identical**
  to `en` (no missing or extra keys). Missing translations fail CI.
- Existing render tests run against an i18n instance initialized with `en`.

## Open questions / assumptions

- **Assumed** scope = frontend UI shell only (Rust strings English in phase 1).
  Confirmed with user.
- **Assumed** library = react-i18next. Confirmed with user.
- Translation authorship (human vs. machine-assisted, review workflow) is an
  implementation-plan detail, not a design constraint.
